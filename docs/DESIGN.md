# CasualMultiplay 休闲多人游戏聚合情报站 设计思路

## 一、为什么做这个

原来 HorrorGameInspector 只盯恐怖撤离一个品类，但市场上合作、竞技、派对、非对称这些休闲多人品类互相交叉，同一款游戏经常同时挂多个标签。只看恐怖会漏掉同类竞品，调研时不得不在 SteamDB、SteamCharts、七麦、TapTap 之间反复跳，效率很低。

所以把页面升级成聚合站，不只盯恐怖，而是把所有休闲多人品类拉到一起看。恐怖变成其中一个标签，和合作、派对、PvP 平级。

## 二、解决什么问题

**第一，品类分散。** 同一个品类下的游戏散落在不同平台，想横向对比得开十几个标签页。聚合后一个页面搞定。

**第二，新品发现靠人肉。** 以前找新品靠刷 Steam 新品区，容易漏。现在用 Steam 标签接口自动发现，按标签组合筛选，系统推什么我看什么。

**第三，数据来源混乱。** 网上各种数字满天飞，哪个可信哪个不可信没法判断。我们定了一套数据源分级，只采信 Steam 官方、SteamDB、七麦这三类权威源，社区参考数据单独标注，不混在一起。

**第四，单品类视角太窄。** 只看恐怖撤离，看不到隔壁合作派对品类的趋势。聚合后可以跨品类对比，发现哪些玩法正在融合。

## 三、页面怎么组织

主导航四项，简单直接。

**发现页** 是入口。进来先看到全部游戏的卡片网格，顶部一排标签筛选条，想看恐怖就选恐怖，想看合作就选合作，可以多选。搜索框可以按名字找。排序按优先级、CCU、愿望单、发售日、好评率都行。卡片点进去就进游戏详情。

**排行榜页** 是横向对比用的。一张表，列是各款游戏，行是不同维度。维度可以切，想看 CCU 排行就切 CCU，想看愿望单排行就切愿望单，想看好评率就切好评率。表格上方标清楚数据来源和更新时间，所有数字都能点进去看原始链接。

**游戏详情页** 是核心。和之前讨论的方案一样，一个页面，左边是游戏列表，右边是详情。点左边不同游戏，右边原地刷新，不跳页面。这样切游戏很快，列表常驻在左边，上下文不丢。

左边列表有搜索、标签筛选、排序三层定位方式。超自然行动组作为普通竞品放在列表里，靠置顶字段排在最前面，和其他游戏样式完全一样，不做特殊分组。

右边详情区从上到下是：游戏名和开发商标签、四个关键指标卡（CCU、好评率、愿望单、发售日）、游戏介绍、近期动态时间线、数据来源和更新时间。

**竞品对比页** 是给汇报用的。选两到四款游戏，并排展示各项指标，高亮最优值。截图就能直接贴进汇报材料。

## 四、数据从哪来

只认三类来源。

**Steam 官方** 是最权威的。CCU、好评率、评测数、发售日、开发商、标签、新闻公告，全部走 Steam 的公开接口，直接采信。

**SteamDB** 是 Steam 社区最权威的第三方。愿望单关注数和排名这里拿，因为 Steam 官方不公开愿望单数字，SteamDB 通过关注数推算，行业普遍认可。

**七麦** 是 iOS 畅销榜和免费榜的来源，直接返回 JSON，不需要逆向。

社区参考数据（Reddit 讨论热度、YouTube 播放量）单独标注，默认折叠，不作为核心指标。

每个数字旁边都有来源标记，绿色是 Steam 官方，蓝色是 SteamDB，灰色是社区参考。鼠标放上去能看到原始链接和抓取时间。

## 五、怎么保证数据可信

定了一套规矩。

**第一，不用来路不明的聚合站。** 网上那种「游戏数据大全」之类的站点一律不采信，因为数据来源不可追溯。

**第二，不用维基和媒体文章里的数字。** 维基可被随意编辑，媒体引用的数字往往没有官方背书。

**第三，每个数字必须带来源链接和抓取时间。** 没有来源的数字不展示。

**第四，数据异常要标记。** 比如 CCU 突然暴涨暴跌，系统会标记异常，保留数据但加警告，不会悄悄覆盖掉。

**第五，数据过期要提示。** 超过 24 小时没更新的数据标灰，提示「数据延迟」，让看的人心里有数。

## 六、怎么维护

纯静态站，没有后端。

所有游戏信息集中在一个 JSON 注册表里，包含游戏名、开发商、标签、优先级、介绍。加新游戏就改这个文件，不用动页面代码。

各平台数据由 GitHub Actions 定时跑爬虫抓回来，存成 JSON，页面加载时读取。爬虫每天早上六点跑一次，重点游戏的 CCU 每小时抓一次。

部署在 GitHub Pages 上，零成本，代码推上去页面自动更新。

## 七、视觉风格

暖灰底色，不刺眼。卡片用白底，边框 0.5px 细线。主交互色用暗红，选中状态用浅红填充。文字两级：标题深近黑，正文中灰。

整体追求安静、克制、信息密度高但不挤。每个数字旁边都有来源标记，让看的人随时知道这个数字从哪来，能信几分。

## 八、分几步做

现在 Phase 1 已完成，项目从 HorrorGameInspector 复制过来，改名重写，注册表建好，仓库推上 GitHub。

接下来 Phase 2 做发现页和排行榜页的标签筛选和搜索。Phase 3 写自动发现脚本，用 Steam 标签接口找新品。Phase 4 接 SteamDB 和七麦的数据源。Phase 5 把详情页从硬编码改成动态生成。Phase 6 做竞品对比页。Phase 7 加虚拟滚动，应对游戏数量超过一百的情况。

每一步做完都能独立用，不依赖后面的步骤。

## 九、核心原则

数据可信比数据完整重要，宁可少展示也不展示不可信的。

能在同一页面用筛选解决的事，不开新页面。

每个数字都能追溯到原始来源。

核心指标优先展示，次要数据折叠。

静态部署优先，降低维护成本。

所有游戏元数据集中管理，不在页面里硬编码。
